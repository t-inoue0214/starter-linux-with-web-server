# 第6章: シェル環境をカスタマイズする

## 前提知識

この章を始める前に、以下の章を完了していること:

- 第2章: 基本コマンドを使いこなす
- 第5章: テキストエディタ3種を使う

---

## 概要

Linux のシェルは、設定ファイルを編集することで自分の作業スタイルに合わせてカスタマイズできます。
この章では `~/.bashrc` にエイリアスとプロンプト（PS1）を追記し、快適なターミナル環境を整えます。

また、第20章でソースビルドする nginx の実行ファイルは `/usr/local/nginx/sbin/nginx` に配置されます。
この章の最後でその場所を `PATH` に追加しておくことで、第20章でスムーズに作業できるようになります。

---

## 手順

### 6-1. bash の設定ファイル体系を知る

bash はシェルが起動するときに、いくつかの設定ファイルを自動的に読み込みます。
どのファイルが読み込まれるかは「ログインシェル」か「非ログインシェル」かで変わります。

#### ログインシェル と 非ログインシェル

| 種類 | 起動のタイミング | 例 |
|:---|:---|:---|
| ログインシェル | SSH でサーバーに接続したとき、`su -` でユーザーを切り替えたとき（`su`: switch user — ユーザーを切り替えるコマンド。第9章で詳しく学ぶ） | `ssh user@server` |
| 非ログインシェル | すでにログイン済みの環境でターミナルを新しく開いたとき | Codespaces のターミナルタブを開く |

**Codespaces のターミナルは非ログインシェルです。**

#### 設定ファイルの読み込み順序

```text
ログインシェル（Login Shell）:
  /etc/profile               ← システム全体（全ユーザー共通）
  /etc/profile.d/*.sh        ← システム全体（モジュール方式）
  ~/.bash_profile            ← ユーザー固有（存在すれば優先）
  ~/.bash_login              ← ~/.bash_profile がない場合
  ~/.profile                 ← ~/.bash_login もない場合
  ※ ~/.bash_profile の中で ~/.bashrc を source するのが一般的

非ログインシェル（Interactive Non-Login Shell）:
  /etc/bash.bashrc           ← システム全体
  ~/.bashrc                  ← ユーザー固有（通常ここを編集する）

ログアウト時:
  ~/.bash_logout             ← ログアウト時に実行される処理
```

#### なぜ `~/.bashrc` を編集するのか

Codespaces（非ログインシェル）では `~/.bash_profile` は読み込まれません。
どちらの環境でも確実に反映させたい設定は `~/.bashrc` に書くのが安全です。

現在の内容を確認してみましょう:

```bash
# 現在の ~/.bashrc の内容を確認
$ cat ~/.bashrc
# ~/.bashrc: executed by bash(1) for non-login shells.
...
# alias ls='ls --color=auto'
#alias ll='ls -l'
#alias la='ls -A'
...
export PROMPT_DIRTRIM=4
（環境によって内容は異なります。この後 vim で末尾に追記していきます）

# システム全体の設定も確認
$ cat /etc/bash.bashrc
# System-wide .bashrc file for interactive bash(1) shells.
...
（システム全体のデフォルト設定が表示されます）
```

### 6-2. エイリアスを設定する

`alias` コマンドを使うと、長いコマンドに短い名前をつけられます。

#### まず一時的に試してみる

```bash
# 一時的なエイリアスを設定（現在のシェルセッションのみ有効）
$ alias ll='ls -la'
$ ll                  # 動作確認
total 8
drwxr-xr-x 2 vscode vscode 4096 May 19 11:44 .
drwxr-xr-x 3 vscode vscode 4096 May 19 11:44 ..
（カレントディレクトリのファイル一覧が詳細表示されます）

# 現在設定されているエイリアスを一覧表示
$ alias
alias grep='grep --color=auto'
alias ll='ls -la'
alias ls='ls --color=auto'
（環境によってすでに定義されているエイリアスが追加で表示されます）
```

#### `~/.bashrc` に追記して永続化する

```bash
$ vim ~/.bashrc
```

ファイルの末尾に以下を追記します:

```bash
# Aliases
alias ll='ls -la'
alias la='ls -A'
alias grep='grep --color=auto'
```

> **`alias ls='ls --color=auto'` について**
> Codespaces のデフォルト `~/.bashrc` には `alias ls='ls --color=auto'` がすでに定義されています（`cat ~/.bashrc` で確認できます）。
> 重複して追記しても動作に問題はありませんが、不要なので追記しなくて構いません。
>
> また、デフォルトの `~/.bashrc` には `#alias ll='ls -l'` と `#alias la='ls -A'` がコメントアウトで存在します。
> コメントを外す方法もありますが、`ll='ls -l'`（隠しファイル非表示）と今回推奨する `ll='ls -la'`（隠しファイル含む）ではオプションが異なります。
> コメントを外すのではなく、末尾に新しく追記してください。

```bash
# 設定を即時反映
$ source ~/.bashrc

# 動作確認
$ ll
total 8
drwxr-xr-x 2 vscode vscode 4096 May 19 11:44 .
drwxr-xr-x 3 vscode vscode 4096 May 19 11:44 ..
（隠しファイルを含む詳細一覧が表示される）

$ la
（ドット始まりの隠しファイルを除いた一覧が表示される。空ディレクトリでは何も表示されない）
```

#### エイリアスを最小限にする理由

`alias ..='cd ..'` や `alias cls='clear'` のように「コマンドを省略するエイリアス」は一見便利ですが、**実務では設定しないことを推奨します。**

本番サーバーには自分のエイリアスが存在しません。
手癖でエイリアスに頼っていると、本番環境で `..` と打ってエラーになるような誤操作を引き起こすリスクがあります。

設定してよいエイリアスは「**出力を見やすくする**」目的のものに限定しましょう:

- `ls --color=auto` → ファイル種別を色で区別
- `grep --color=auto` → マッチした箇所を色でハイライト
- `ll='ls -la'` → 詳細一覧表示（よく使う組み合わせの短縮）

基本コマンド（`cd ..`・`clear` 等）はそのまま覚えることを推奨します。

### 6-3. プロンプト（PS1）をカスタマイズする

ターミナルの入力プロンプト（`vscode@host:~$` のような表示）は `PS1` という変数で制御されています。

```bash
# 現在の PS1 を確認
$ echo $PS1
```

#### 主要なエスケープシーケンス

> **「エスケープシーケンス」とは?**
> `\u` や `\h` のように、バックスラッシュ（`\`）で始まる特殊な文字列のことです。
> PS1 の中でこれらが使われると、bash がプロンプト表示時にそれぞれの値に展開します。
> 例: `\u` → `vscode`、`\h` → `codespaces-abc123`

| 記号 | 表示内容 |
|:---|:---|
| `\u` | ユーザー名 |
| `\h` | ホスト名（短縮形） |
| `\H` | ホスト名（FQDN = 完全修飾ドメイン名。`hostname.example.com` のようなドット付きフル表記） |
| `\w` | カレントディレクトリ（**フルパス**） |
| `\W` | カレントディレクトリ（末尾のディレクトリ名のみ） |
| `\$` | `$`（一般ユーザー）または `#`（root） |
| `\n` | 改行 |
| `\[\e[32m\]` | 色開始（緑）|
| `\[\e[34m\]` | 色開始（青）|
| `\[\e[0m\]` | 色リセット |

#### `\u@\h`（ユーザー名＋ホスト名）の重要性

「ユーザー名とホスト名はプロンプトに必ず表示する」ことを強く推奨します。

開発・検証・本番と複数のサーバーを SSH で操作する場面では、**今どのサーバーにいるかがひと目でわかること**が誤操作防止の基本です。
本番に複数台のサーバーがある場合も同様です。

慣れてくるとプロンプトの情報量を減らしたくなることがありますが、本番 SSH の機会が増えてきたら必ず `\u@\h` を含めることを習慣にしてください。

#### `\$` の意義（権限の常時確認）

- `$` → 一般ユーザーとして操作中
- `#` → **root として操作中**

`sudo -i` や `su -` で root に切り替えたとき、プロンプトが `#` に変わることで「今は root権限で操作している」と視覚的に確認できます。権限を意識したまま操作する習慣がミスを防ぎます。

#### `\w`（フルパス）を推奨する理由

`\W`（末尾のディレクトリ名のみ）より `\w`（フルパス）を推奨します。

- `scp` でファイル転送するとき、フルパスをそのまま使えて便利
  - 例: プロンプトに `/etc/nginx/conf.d` と表示されていれば `scp user@host:/etc/nginx/conf.d/default.conf .` がすぐ書ける
- ターミナルの操作記録を見返すときにどこで何をしていたかが一目でわかる

#### 改行プロンプト（推奨形式）

フルパスが長いディレクトリにいると、入力開始位置がパスの長さによって変わり操作しづらくなります。
`\n` で改行し、`\$` を次の行に置くことで**入力位置が常に行頭**になります。

```bash
PS1='\[\e[32m\]\u@\h\[\e[0m\]:\[\e[34m\]\w\[\e[0m\]\n\$ '
```

表示イメージ:

```text
vscode@codespaces-abc123:/etc/nginx/conf.d
$ 
```

どんなに深いディレクトリにいても、入力は常に `$` の後から始まります。

#### `\[` と `\]` が必要な理由

色コードなどの「画面に表示されない文字」は、`\[` と `\]` で囲む必要があります。

bash はプロンプトの「表示幅」を計算してカーソル位置を管理しています。
色コードは見えない文字ですが、囲まないと幅の計算に含まれてしまい、長い行を入力したときに表示が崩れます。

#### 実習: 推奨 PS1 を `~/.bashrc` に追記する

```bash
$ vim ~/.bashrc
```

先ほど追記したエイリアスの下に続けて追記します:

```bash
# Custom prompt
unset PROMPT_DIRTRIM
PS1='\[\e[32m\]\u@\h\[\e[0m\]:\[\e[34m\]\w\[\e[0m\]\n\$ '
```

> **`unset PROMPT_DIRTRIM` が必要な理由**
> Codespaces のデフォルト `~/.bashrc` には `export PROMPT_DIRTRIM=4` が設定されています。
> これは `\w` の表示を末尾4コンポーネントに切り詰める設定です。
> この設定が有効なまま `\w` を使うと、深いディレクトリにいるとフルパスが表示されません。
> `unset PROMPT_DIRTRIM` でこの制限を解除することで、`\w` が本来のフルパスを表示します。

```bash
# 反映して確認
$ source ~/.bashrc
```

プロンプトが色付きになり、改行されて `$` から入力できるようになれば成功です。

### 6-4. source コマンドを理解する

`source` は設定ファイルを**現在のシェルで実行**するコマンドです。

```bash
source ~/.bashrc     # 現在のシェルで実行 → 環境変数・エイリアスが即時反映
. ~/.bashrc          # ドットスペース — source と同じ意味（POSIX 標準の書き方）
```

#### なぜ新しいターミナルを開かないと反映されないのか

`~/.bashrc` を編集しただけでは、**現在起動しているシェルには影響しません。**

シェルは起動時に一度だけ設定ファイルを読み込みます。その後は「読み込み済みの状態」で動き続けるため、ファイルを変更しても自動的には反映されません。

`source ~/.bashrc` を実行すると、そのファイルを今すぐ現在のシェルで読み直します。
新しいターミナルを開くと、起動時に自動的に `~/.bashrc` が読まれるため、そちらでも反映されます。

#### `source` と `bash スクリプト実行` の違い

```bash
source ~/.bashrc     # 現在のシェルで実行 → export した変数が現在のシェルに残る
bash ~/.bashrc       # 子シェルで実行   → 子シェルが終了すると変数は消える
```

`bash ~/.bashrc` は「子プロセスを作って実行する」ため、そこで `export` した変数は現在のシェルには引き継がれません。設定を反映させたいときは必ず `source` を使います。

### 6-5. 第20章への前準備：nginx の PATH を追加する

> **「PATH」とは?**
> シェルが「コマンド名だけで実行できるディレクトリ」のリストを保持する環境変数です。
> `ls` と打ったとき、シェルは PATH の各ディレクトリを左から順番に探し、最初に見つかった `/usr/bin/ls` を実行します。
> `echo $PATH` で確認できます。

第20章でソースビルドした nginx の実行ファイルは `/usr/local/nginx/sbin/nginx` に配置されます。
今のうちに `PATH` へ追加しておくことで、第20章でフルパス指定なしに `nginx` コマンドが使えるようになります。

```bash
$ vim ~/.bashrc
```

ファイルの末尾に追記します:

```bash
# PATH for nginx source build (used in chapter-20)
export PATH="/usr/local/nginx/sbin:$PATH"
```

```bash
# 反映して確認
$ source ~/.bashrc
$ echo $PATH
/usr/local/nginx/sbin:/vscode/vscode-server/bin/.../remote-cli:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin:...
```

`/usr/local/nginx/sbin` が先頭に追加されていれば成功です（まだディレクトリは存在しませんが問題ありません）。
Codespaces 環境では `/vscode/vscode-server/...` など VS Code 関連のパスも含まれるため、実際の出力は長くなります。

> **`:$PATH` を必ず書く理由**
> `export PATH="/usr/local/nginx/sbin"` と書いてしまうと、**既存の PATH が消えます。**
> `/usr/bin` や `/bin` がなくなるため `ls`・`cat` など基本コマンドが "command not found" になります。
> 末尾の `:$PATH` は「既存の PATH に追加する」という意味です。

> **存在しないディレクトリを PATH に入れても問題ない**
> `PATH` に追加したディレクトリが存在しない場合、そこにコマンドが見つからないだけで、他のコマンドには影響しません。
> 第20章でソースビルドが完了すると、自動的にそのパスが有効になります。

## よくあるミス

| ミス | 内容 | 対処 |
|:---|:---|:---|
| `~/.bashrc` 編集後に `source` を忘れる | 新しいターミナルを開くまで設定が反映されない | `source ~/.bashrc` または `. ~/.bashrc` を実行する |
| `~/.bash_profile` と `~/.bashrc` を混同 | Codespaces（非ログインシェル）では `~/.bash_profile` は読まれない | 通常は `~/.bashrc` を編集する |
| PS1 の `\[` と `\]` を省略する | カーソル位置がずれて、長い行の入力時に表示が崩れる | 色コードは必ず `\[...\]` で囲む |
| `export PATH="/usr/local/nginx/sbin"` と書く | 既存の PATH が消えて基本コマンドが使えなくなる | 末尾に `:$PATH` を必ず追加する |

## 類似比較

| 設定の種類 | 書き方の例 | 適したユースケース |
|:---|:---|:---|
| エイリアス | `alias ll='ls -la'` | 長いコマンドを短い名前で登録する |
| シェル関数 | `mkcd() { mkdir -p "$1" && cd "$1"; }` | 引数を受け取る複合操作 |
| 環境変数 | `export EDITOR='vim'` | 他のプログラムから参照される設定値 |
| PATH 追加 | `export PATH="/usr/local/nginx/sbin:$PATH"` | 新しいコマンドの置き場所を検索対象に加える |
| PS1 カスタマイズ | `PS1='\u@\h:\w\$ '` | プロンプトの表示形式を変更する |

## 他OSとの比較

| 設定項目 | Linux (bash) | Windows (PowerShell) | macOS (zsh) |
|:---|:---|:---|:---|
| ユーザー設定ファイル | `~/.bashrc` | `$PROFILE` | `~/.zshrc` |
| システム全体の設定 | `/etc/bash.bashrc` | — | `/etc/zshrc` |
| ログイン時の設定 | `~/.bash_profile` | — | `~/.zprofile` |
| 即時反映 | `source ~/.bashrc` | `. $PROFILE` | `source ~/.zshrc` |

## 理解度チェック

1. Codespaces のターミナルはログインシェルと非ログインシェルのどちらか？読み込まれる設定ファイルはどれか？

<details><summary>答え</summary>

**非ログインシェル**です。
非ログインシェルでは `/etc/bash.bashrc` と `~/.bashrc` が読み込まれます。
`~/.bash_profile` は読み込まれないため、Codespaces での設定変更は `~/.bashrc` に書きます。

</details>

2. `~/.bashrc` を編集した後、現在のターミナルに即時反映させるコマンドは？

<details><summary>答え</summary>

```bash
source ~/.bashrc
```

または同じ意味の:

```bash
. ~/.bashrc
```

</details>

3. `alias ll='ls -la'` をターミナルで直接実行した後、新しいターミナルを開いたら `ll` が使えなかった。理由と対処を説明せよ。

<details><summary>答え</summary>

**理由**: コマンドラインで設定したエイリアスは、現在のシェルセッションのみで有効です。新しいターミナルは別のシェルプロセスとして起動するため、引き継がれません。

**対処**: `~/.bashrc` にエイリアスを追記して `source ~/.bashrc` で反映させます。これで新しいターミナルを開いても有効になります。

</details>

4. PS1 の色コードを `\[` と `\]` で囲む必要があるのはなぜか？

<details><summary>答え</summary>

bash はプロンプトの**表示幅**を計算してカーソル位置を管理しています。
色コードは画面に表示されない文字ですが、`\[...\]` で囲まないと幅の計算に含まれてしまいます。
その結果、長い行を入力して折り返すタイミングがずれ、表示が崩れます。
`\[` と `\]` で囲むことで「この範囲は幅ゼロの文字列」として扱われます。

</details>

5. `export PATH="/usr/local/nginx/sbin:$PATH"` の末尾の `:$PATH` を省略するとどうなるか？

<details><summary>答え</summary>

既存の `PATH`（`/usr/bin`・`/bin` など）が**完全に上書きされて消えます**。
`ls`・`cat`・`grep` など基本的なコマンドが "command not found" になります。

`:$PATH` は「既存の PATH の内容を引き継ぎながら先頭に追加する」という意味です。
PATH を変更するときは必ず `:$PATH` を末尾に書く習慣をつけてください。

</details>
