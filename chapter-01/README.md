# 第1章: シェルの種類と選び方

## 前提知識

この章を始める前に、以下の章を完了していること:

- 第0章: Linux の歴史・ディストリビューションとは

---

## 概要

シェルは、ユーザーが入力したコマンドを受け取り Linux カーネルに渡す「橋渡し役」のプログラムです。
bash・sh・zsh など複数の種類があり、特に `#!/bin/bash` と `#!/bin/sh` の違いや、
`./deploy.sh` / `sh deploy.sh` / `source deploy.sh` の使い分けは現場でよく混乱するポイントです。
この章ではシェルの基本を理解し、スクリプトの起動方法を実際に手を動かして習得します。

---

> **コマンド例の読み方:**
> このドキュメントのコマンド例では行頭に `$` が付いています。
> これは「ここがコマンドを入力する場所ですよ」を示すプロンプト記号で、**`$` 自体は入力しません**。
> Windows の `C:\Users\username>` や `C:\>` に相当します。
> `$` に続く部分だけを入力して Enter を押してください。

## 手順

### 1-1. シェルとは何か

**ターミナルとシェルは別物です:**

| | 役割 | 例 |
|:---|:---|:---|
| **ターミナル**（端末エミュレータ） | 文字を入出力する「画面の枠」 | VS Code のターミナルパネル、gnome-terminal |
| **シェル** | ターミナルの中で動くプログラム。コマンドを解釈してカーネルに渡す | bash, zsh, sh |

VS Code でターミナルを開いたとき、パネル自体が「ターミナル」、その中で動いている bash が「シェル」です。

**Windows ユーザーへ:** Windows の「コマンドプロンプト」は、ターミナルとシェルが一体化したものです。
Linux では画面の枠（ターミナル）と中で動くプログラム（シェル）が分かれており、組み合わせを自由に選べます。
現在は Windows でも「Windows Terminal」という別アプリが登場し、cmd.exe と PowerShell の両方をタブで切り替えて使えるようになっています。この事実からも、ターミナルとシェルは別物であることが分かります。

| | Windows | Linux（Codespaces） |
|:---|:---|:---|
| 画面の枠（ターミナル） | コマンドプロンプト / Windows Terminal | VS Code のターミナルパネル（gnome-terminal 等） |
| コマンドを解釈するプログラム（**シェル**） | **cmd.exe** / PowerShell | **bash** / zsh / sh |
| 命令を書いたテキストファイル（**スクリプト**） | **バッチファイル**（`.bat`） | **シェルスクリプト**（`.sh`） |
| 普段「〇〇を開く」と言うもの | 「コマンドプロンプトを開く」= ターミナル+シェルが同時に起動 | 「ターミナルを開く」= ターミナルが起動し、中でシェル（bash）が起動 |

> **「Windowsバッチ」と「シェル」の関係:**
>
> - `cmd.exe` = bash に相当する「**シェル（プログラム）**」
> - `.bat` ファイル = `.sh` ファイルに相当する「**スクリプト（テキストファイル）**」
>
> 「Windowsバッチ」と呼ばれる `.bat` ファイルは bash スクリプト（`.sh`）と対になります。
> その `.bat` を実行するプログラムが `cmd.exe`（= Windows のシェル）です。

```text
ユーザー（コマンドを入力）
      ↓ ターミナルが文字を受け取る
 シェル（bash / zsh 等）
      ↓ コマンドを解釈して実行命令に変換
 Linux カーネル
      ↓
 ハードウェア（CPU・メモリ等）
```

**sh（Bourne Shell）と Unix の関係:**

第0章で Unix（1969年）が Linux の親にあたることを学びました。
`sh`（Bourne Shell）は Unix 時代（1979年）から存在する最も古いシェルです。
現在の Linux で使われる bash・zsh 等はすべてこの `sh` の流れを汲んでいます。
Debian では `/bin/sh` は `dash` という軽量シェルへのシンボリックリンクになっています（後の手順で確認します）。

### 1-2. 現在のシェルを確認する

```bash
$ echo $SHELL
/bin/bash
```

```bash
$ cat /etc/shells
# /etc/shells: valid login shells
/bin/sh
/usr/bin/sh
/bin/bash
/usr/bin/bash
/bin/rbash
/usr/bin/rbash
/usr/bin/dash
/bin/zsh
/usr/bin/zsh
```

- `$SHELL`: ログイン時に起動するデフォルトシェルのパスが格納された環境変数
- `/etc/shells`: このシステムで「ログインシェル（ログイン時、最初に起動されるシェル）」として設定できるシェルの一覧ファイル

> **`rbash` とは?** 一覧に含まれる `rbash` は「restricted bash（制限付き bash）」です。
> 特定のコマンドしか実行できないよう制限された特殊なシェルで、通常の開発では使いません。

### 1-4. シェバン行（1行目のおまじない）を理解する

シェルの例）

```bash
#!/bin/bash

echo Hello!
```

シェルスクリプトの1行目には、**このスクリプトをどのシェルで実行するかを OS に伝える**行を書きます。
これを**シェバン行**（shebang、シバンとも）と呼びます。

```bash
#!/bin/bash   ← bash で実行する
#!/bin/sh     ← sh（Debian では dash）で実行する
```

**`#!/bin/bash` と `#!/bin/sh` の違い:**

| シェバン行 | 実行されるシェル（Debian） | 使うべき場面 |
|:---|:---|:---|
| `#!/bin/bash` | bash（機能豊富） | bash 固有機能（配列・`[[` 条件式等）を使うとき |
| `#!/bin/sh` | dash（`/bin/sh` の実体） | どの Linux でも動く汎用スクリプトを書くとき |

実際に `/bin/sh` の実体を確認してみましょう:

```bash
$ ls -la /bin/sh
lrwxrwxrwx 1 root root 4 Feb  4  2025 /bin/sh -> dash
```

`/bin/sh` は `dash` へのシンボリックリンク（**シンボリックリンク**: Windows のショートカット（.lnk ファイル）に相当する「別名でファイルを参照する仕組み」。第3章で詳しく学びます）です。

> **よくある落とし穴:** シェバン行が `#!/bin/bash` で、bash 固有の構文を使ったスクリプトを
> `sh deploy.sh` で実行するとエラーになります（`sh` は bash 固有機能を知らないため）。

---

### 1-5. スクリプトの実行方法の違い（`./` vs `sh` vs `source`）

同じスクリプトファイルでも、**起動方法によって動作が異なります**。現場でよく混乱するポイントです。

まず確認用スクリプトを作成します:

> **`/tmp` について:** 一時ファイルを置くためのディレクトリです。Windows の `C:\Temp`（`%TEMP%`）
> に相当します。再起動すると削除されることがあります。今回はここに練習用スクリプトを作成します。

```bash
$ cat > /tmp/test.sh << 'EOF'
#!/bin/bash
echo "実行シェル: $0"
echo "MY_VAR の値: ${MY_VAR}"
MY_VAR="スクリプト内で変更"
echo "変更後: ${MY_VAR}"
EOF
```

> **`<< 'EOF'` ～ `EOF` について:** 複数行のテキストをそのままファイルに書き込む記法（ヒアドキュメント）です。
> 詳しくは第2章で学びます。今は「複数行をまとめてファイルに書き込めるおまじない」として使ってください。

```bash
$ export MY_VAR="元の値"
$ echo $MY_VAR
元の値
```

> **`export` について:** 変数を「現在のシェルだけでなく、そこから起動するプログラムにも引き渡す」コマンドです。
> 環境変数については第7章で詳しく学びます。

**① `./test.sh` — シェバン行で指定したシェルが別プロセスとして実行**

```bash
$ chmod +x /tmp/test.sh   # chmod: ファイルのパーミッション（読み書き実行の権限）を変更するコマンド。+x で実行可能にする（詳細は第11章）
$ cd /tmp
$ ./test.sh
実行シェル: ./test.sh
MY_VAR の値: 元の値
変更後: スクリプト内で変更
$ echo $MY_VAR
元の値         ← 現在のシェルの MY_VAR は変わっていない
```

> **`./` とは?** 「カレントディレクトリ（今いるフォルダ）にあるファイルを実行する」という意味です。
> `cd /tmp` で `/tmp` に移動したので、`./test.sh` は `/tmp/test.sh` と同じファイルを指します。

**② `sh test.sh` — 明示的に sh（dash）で実行（別プロセス）**

```bash
$ sh /tmp/test.sh
実行シェル: /tmp/test.sh
MY_VAR の値: 元の値
変更後: スクリプト内で変更
```

**③ `source test.sh` — 現在のシェルの中で実行（別プロセスを作らない）**

```bash
$ source /tmp/test.sh
実行シェル: bash
MY_VAR の値: 元の値
変更後: スクリプト内で変更
$ echo $MY_VAR
スクリプト内で変更   ← 現在のシェルの MY_VAR が書き換わった！
```

**3つの実行方法の違いまとめ:**

> **「プロセス」とは?** プログラムの実行単位です。Windows のタスクマネージャー（Ctrl+Shift+Esc）を
> 開いたときに一覧表示される各アプリケーション・プログラムがそれぞれ1つのプロセスです。
> `./test.sh` を実行すると bash のプロセスが新しく1つ起動し、スクリプトが終わると消えます。

| 実行方法 | 別プロセスを作るか | 実行されるシェル | スクリプト内の変数が現在シェルに反映されるか |
|:---|:---|:---|:---|
| `./test.sh` | 作る | シェバン行で指定したシェル | されない |
| `sh test.sh` | 作る | sh（dash） | されない |
| `bash test.sh` | 作る | bash | されない |
| `source test.sh`（`. test.sh` も同じ） | 作らない | 現在のシェル | **される** |

> **`~`（チルダ）について:** `~` はホームディレクトリ（ログインユーザーの個人フォルダ）を表す略記号です。
> Windows の `C:\Users\yourname`（`%USERPROFILE%`）に相当します。
> `~/.bashrc` は `/home/ユーザー名/.bashrc` と同じ意味です。

> **`source` を使う典型例:** `~/.bashrc` を編集した後、その変更を今のターミナルに反映するために
> `source ~/.bashrc` を実行します。`bash ~/.bashrc` では別プロセスで実行されるため、
> 現在のシェルには変更が反映されません。

### 1-6. デフォルトシェルを変更する

```bash
$ chsh -s /bin/bash
Password:
```

> **[コンテナ制限] Codespaces 環境での注意**
> この操作は GitHub Codespaces (Docker コンテナ) 上では動作が制限される場合があります。
> `chsh` を実行しても、次回接続時に反映されないことがあります。
> また、Codespaces では最初から bash がデフォルトなので、この講座の実習では `chsh` を実行しなくても問題ありません。
> 変更を確認するには再ログインが必要です: `echo $SHELL`

## よくあるミス

| ミス | エラーメッセージ例 | 正しい対処 |
|:---|:---|:---|
| `./deploy.sh` で実行権限エラー | `bash: ./deploy.sh: Permission denied` | `chmod +x deploy.sh` で実行権限を付ける（第11章で詳しく学ぶ） |
| `#!/bin/bash` のスクリプトを `sh` で実行してエラー | `sh: 1: [[: not found` 等 | `bash deploy.sh` で実行する。または `#!/bin/sh` に書き換えて bash 固有構文を除く |
| `source` と `./` を混同して環境変数が反映されない | —（エラーなし。変数が変わらないだけ） | 環境変数を現在のシェルに反映したいなら `source`、そうでなければ `./` でよい |
| `chsh` 後にすぐシェルが変わると思う | —（エラーなし。変わっていないだけ） | 再ログインが必要。`echo $SHELL` で確認 |

## 類似比較

| シェル | bash スクリプトは動くか | 主な採用環境 | 特徴 |
|:---|:---|:---|:---|
| **bash** | ◎（自分自身） | Linux 全般のデフォルト | 最普及。スクリプトの事実上の標準 |
| sh / dash | △（bash 固有機能は NG） | システム起動スクリプト（`/bin/sh`） | 軽量・高速。Unix 互換の最小仕様のみ |
| zsh | ◎（bash とほぼ互換） | macOS Catalina 以降デフォルト | bash 互換 + 豊富な補完・プラグイン |
| fish | ✗（構文が大きく異なる） | デスクトップ用途 | 自動補完・カラーが優秀。スクリプト互換性は低い |

> **POSIX（ポジックス）とは:** Unix 系 OS が守るべき共通仕様のルール集（IEEE 規格）。
> 「POSIX 準拠」のシェルスクリプトはほとんどの Unix/Linux 環境で動きます。
> bash は POSIX を超える独自拡張を持ちますが、sh/dash は POSIX の最小仕様に徹しています。

## 他OSとの比較

| 観点 | Linux（Debian） | Windows | macOS |
|:---|:---|:---|:---|
| デフォルトシェル | bash | cmd.exe（bash に相当する Windows のシェル）/ PowerShell | zsh（Catalina 以降） |
| `/bin/sh` 相当 | dash（`/bin/sh -> dash`） | — | sh（`/bin/sh -> bash` のことが多い） |
| スクリプト拡張子 | `.sh`（bash スクリプト） | `.bat`（cmd）/ `.ps1`（PowerShell） | `.sh` |

| シェル設定ファイル | `~/.bashrc`、`~/.bash_profile` | — | `~/.zshrc` |

## 理解度チェック

1. ターミナルとシェルの違いを説明してください。VS Code でターミナルを開いたとき、「ターミナル」と「シェル」はそれぞれ何に相当しますか？

<details><summary>答え</summary>

- **ターミナル（端末エミュレータ）**: 文字を入出力する「画面の枠」。VS Code のターミナルパネルがこれにあたる
- **シェル**: ターミナルの中で動くプログラム。コマンドを解釈して Linux カーネルに渡す役割を持つ

VS Code でターミナルを開いたとき:

- 「ターミナル」= VS Code のターミナルパネル（画面の枠）
- 「シェル」= その中で動いている bash プロセス

「ターミナルを開く = シェルが起動する」という関係。ターミナルを閉じるとシェルも終了する。

</details>

2. シェルスクリプトの1行目に `#!/bin/bash` と書いた場合と `#!/bin/sh` と書いた場合で、Debian 環境での動作はどう異なりますか？

<details><summary>答え</summary>

| シェバン行 | 実行されるシェル（Debian） | 影響 |
|:---|:---|:---|
| `#!/bin/bash` | bash（`/bin/bash`） | bash 固有機能（配列・`[[` 条件式等）が使える |
| `#!/bin/sh` | dash（`/bin/sh -> dash`） | bash 固有機能は使えない。他の Linux でも動く汎用スクリプトになる |

Debian では `/bin/sh` は `dash` へのシンボリックリンク（`ls -la /bin/sh` で確認可能）。
`#!/bin/bash` で書いたスクリプトを `sh deploy.sh` で実行すると、dash が bash 固有構文を解釈できずエラーになる。

</details>

3. `./deploy.sh` と `source deploy.sh` では、スクリプト内で設定した環境変数の扱いが異なります。どう違うか、理由も含めて答えてください。

<details><summary>答え</summary>

- **`./deploy.sh`**: 別プロセスを作って実行する。スクリプト内で設定した変数は別プロセスの中にあるため、現在のシェルには反映されない
- **`source deploy.sh`**: 別プロセスを作らず、現在のシェルの中でスクリプトを実行する。スクリプト内で設定した変数が現在のシェルに反映される

理由: プロセスは「自分の変数の空間（環境）」を独立して持つ。`./` で起動した子プロセスの変数は、親プロセス（現在のシェル）には戻ってこない。`source` は子プロセスを作らないので変数がそのまま現在のシェルに残る。

</details>

4. `source ~/.bashrc` はよく使うコマンドです。`bash ~/.bashrc` ではなく `source` を使う理由を答えてください。

<details><summary>答え</summary>

`~/.bashrc` を編集しても、そのままでは**現在開いているターミナルのシェルには変更が反映されない**。
`~/.bashrc` は新しいシェル起動時に自動で読み込まれるファイルです。そのため既存のシェルには変更が反映されません。

- `bash ~/.bashrc` → 別の bash プロセスで実行される → 現在のシェルには反映されない
- `source ~/.bashrc` → 現在のシェルの中で実行される → コマンドの別名設定（エイリアス、第6章で学ぶ）・環境変数の変更がすぐに反映される

`source`（または `. ~/.bashrc`）を使うことで、ターミナルを開き直さずに設定変更を反映できる。

</details>

5. Debian では `/bin/sh` の実体が `dash` になっています。その事実を確認するコマンドを答えてください。また、bash ではなく dash が選ばれている理由を答えてください。

<details><summary>答え</summary>

**確認コマンド:**

```bash
$ ls -la /bin/sh
lrwxrwxrwx 1 root root 4 Feb  4  2025 /bin/sh -> dash
```

シンボリックリンク（→ の右側）が `dash` を指しているので、`/bin/sh` の実体が dash だとわかる。

**bash ではなく dash が選ばれている理由:**

| 観点 | 内容 |
|:---|:---|
| 軽量・高速 | dash は bash より起動が速く、OS 起動時に大量に呼ばれる `/etc/init.d/` スクリプトの実行を高速化できる |
| POSIX 準拠 | dash は POSIX の最小仕様に徹しているため、bash 固有機能への依存を防げる |
| 安全性 | 機能を絞ることで bash 固有のセキュリティ問題（Shellshock 等）の影響を受けにくい |

このため Debian は「汎用の `/bin/sh`」に bash ではなく dash を採用している。
スクリプトを `#!/bin/sh` で書くと、bash 固有機能が使えない代わりに、どの Linux でも動く汎用スクリプトになる。

</details>
