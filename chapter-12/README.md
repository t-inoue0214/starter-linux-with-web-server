# 第12章: ネットワーク基礎（IP・サブネット・DNS）

## 前提知識

この章を始める前に、以下の章を完了していること:

- 第4章: パッケージ管理 ★ Nginx 初体験（apt）

---

## 概要

第4章で `apt install nginx` を実行し、ブラウザから nginx のページを確認しました。あのとき、裏では何が起きていたのでしょうか？

- ブラウザはどうやってこの Codespaces マシンを見つけたのか？（**IP アドレス**）
- なぜ「ポート 80」という数字が関係するのか？（**ポート**）
- URL の `localhost` はどうやって IP アドレスに変換されるのか？（**DNS**）

この章でこれらの疑問に答えます。ネットワークの知識はサーバーエンジニアにとっての「地図の読み方」です。地図が読めれば、どこで迷子になっているかがわかります。

## 手順

### 12-1. IP アドレスとは何か

#### 住所としての IP アドレス

インターネット上のすべてのマシン（サーバー・スマホ・PC）には「住所」が割り振られています。これが IP アドレスです。

```text
あなたのスマホ                    Web サーバー
 [ブラウザ]  ──────────────▶  [10.0.12.216]
               「この住所に届けて」
```

IP アドレスは 0〜255 の数字を4つ並べた形式です。

```text
10  .  0  .  12  .  216
↑      ↑      ↑      ↑
4つの数字をドットで区切る（合計32ビット）
```

#### プライベートアドレスとパブリックアドレス

| 種類 | 範囲 | 使われる場所 |
|:---|:---|:---|
| プライベート | 10.0.0.0〜10.255.255.255 | 会社・データセンター内部 |
| プライベート | 172.16.0.0〜172.31.255.255 | 会社・データセンター内部 |
| プライベート | 192.168.0.0〜192.168.255.255 | 家庭の WiFi |
| パブリック | 上記以外 | インターネット上で一意の住所 |

> **これらの範囲は「国際標準（RFC 1918）」で決まっています**
> 「なぜ 192.168.x.x が家庭 WiFi なの?」と疑問に思うかもしれません。
> 実は、このアドレス帯は国際的なインターネット標準文書（RFC = Request for Comments）で
> 「プライベートアドレスとして使って良い範囲」と定められています。
> 世界中の WiFi ルーター・データセンターが同じルールに従っているので、
> どこに行っても `192.168.x.x` は「ローカルネットワーク内」を意味します。

Codespaces の IP（10.0.x.x）はプライベートアドレスです。Azure のデータセンター内部専用の住所のため、インターネットには直接出ません。

#### ループバックアドレス（127.0.0.1）

`127.0.0.1` は「自分自身」を指す特別な住所です。手紙を自分宛てに出すようなもので、常に自分のマシンに届きます。`localhost` という名前でも参照できます。

`127.0.0.0/8`（127 で始まる全アドレス）はループバック専用として国際標準（RFC 5735）で予約されています。`127.0.0.1` がその代表例です。なぜ `127.0.0.1` なのか特別な理由はなく、「慣例的にずっとこの値が使われてきた」デファクトスタンダードです。

```bash
$ ping -c 2 127.0.0.1
PING 127.0.0.1 (127.0.0.1) 56(84) bytes of data.
64 bytes from 127.0.0.1: icmp_seq=1 ttl=64 time=0.030 ms
64 bytes from 127.0.0.1: icmp_seq=2 ttl=64 time=0.034 ms
# 0% packet loss = 必ず自分自身に届く
```

#### DHCP — IP アドレスは誰が決める?

スマホで WiFi に繋ぐと、IP アドレスの設定を手動でやった記憶はないはずです。これは DHCP（Dynamic Host Configuration Protocol）という仕組みが自動でやってくれているからです。

```text
スマホ     ───「IP アドレスをください」───▶  WiFi ルーター（DHCP サーバー）
スマホ     ◀───「192.168.1.10 を使って」───  WiFi ルーター
```

Codespaces も同じで、起動時に Azure の DHCP から `10.0.12.216` のような IP を自動でもらっています。

### 12-2. サブネットマスクと CIDR

IP アドレスには「この範囲が同じネットワーク」という区切りがあります。それを表すのが CIDR（サイダー）表記です。

`10.0.12.216/16` の `/16` を読む:

```text
10.0.12.216 / 16
             ↑
             ネットワーク部のビット数
```

`/16` = 先頭 16 ビット（`10.0`）がネットワーク名、残り 16 ビットが個別の番地です。

```text
10.0.  12 .216
└─────┘ └──────┘
ネットワーク  個別番地
（同じ区）   （家の番地）
```

| CIDR | サブネットマスク | 同じネットワークのマシン数 | たとえ |
|:---|:---|:---|:---|
| `/8` | 255.0.0.0 | 約1670万台 | 大規模データセンター |
| `/16` | 255.255.0.0 | 約65,000台 | Codespaces の eth0 |
| `/24` | 255.255.255.0 | 254台 | 家庭の WiFi によくある設定 |
| `/32` | 255.255.255.255 | 1台 | 特定の1台だけを指定 |

### 12-3. 自分の IP アドレスを確認する

#### ip addr — ネットワークインターフェースを確認

```bash
$ ip addr
1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536 qdisc noqueue state UNKNOWN
    link/loopback 00:00:00:00:00:00 brd 00:00:00:00:00:00
    inet 127.0.0.1/8 scope host lo
2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc mq state UP
    link/ether 00:0d:3a:09:36:08 brd ff:ff:ff:ff:ff:ff
    inet 10.0.12.216/16 metric 100 brd 10.0.255.255 scope global eth0
3: docker0: <NO-CARRIER,BROADCAST,MULTICAST,UP> mtu 1500
    link/ether 02:42:9d:9c:dc:a1 brd ff:ff:ff:ff:ff:ff
    inet 172.17.0.1/16 brd 172.17.255.255 scope global docker0
```

> **実際の IP アドレスは環境ごとに異なります**
> `10.0.12.216` は例です。`inet` の行に書かれた IP が各自の Codespaces 環境の IP です。

3つのインターフェース（ネットワーク接続口）が表示されます。

| インターフェース | IP | 役割 |
|:---|:---|:---|
| `lo` | 127.0.0.1 | ループバック（自分自身との通信） |
| `eth0` | 10.0.12.216/16 | Azure ネットワーク（外部との通信） |
| `docker0` | 172.17.0.1/16 | Docker 用ブリッジ（第22章で使う） |

`state UP` = 接続中、`state DOWN` / `NO-CARRIER` = 未接続

#### ip route — パケットはどこへ向かうか

```bash
$ ip route
default via 10.0.0.1 dev eth0 proto dhcp src 10.0.12.216 metric 100
10.0.0.0/16 dev eth0 proto kernel scope link src 10.0.12.216 metric 100
172.17.0.0/16 dev docker0 proto kernel scope link src 172.17.0.1 linkdown
```

`default via 10.0.0.1` = デフォルトゲートウェイ。「宛先が不明なパケットは全部 10.0.0.1 に送る」という意味です。

```text
このマシン                  デフォルトゲートウェイ         インターネット
[10.0.12.216] ─────────▶ [10.0.0.1（玄関）] ─────────▶ 外の世界
```

マンションに例えると、デフォルトゲートウェイはマンションのエントランスです。外に出るときは必ずここを通ります。

### 12-4. 疎通確認（ping）

ping は「あなたはそこにいますか?」と確認するコマンドです。相手から返事（応答）が返ってくれば、ネットワーク的に繋がっています。

```bash
$ ping -c 4 127.0.0.1
PING 127.0.0.1 (127.0.0.1) 56(84) bytes of data.
64 bytes from 127.0.0.1: icmp_seq=1 ttl=64 time=0.030 ms
64 bytes from 127.0.0.1: icmp_seq=2 ttl=64 time=0.034 ms
64 bytes from 127.0.0.1: icmp_seq=3 ttl=64 time=0.031 ms
64 bytes from 127.0.0.1: icmp_seq=4 ttl=64 time=0.033 ms

--- 127.0.0.1 ping statistics ---
4 packets transmitted, 4 received, 0% packet loss, time 3076ms
rtt min/avg/max/mdev = 0.030/0.032/0.034/0.001 ms
```

`-c 4` は 4 回だけ送って終了するオプションです（付けないと無限に送り続けます）。

自分の IP アドレスへも確認できます。

```bash
$ ip addr show eth0 | grep "inet "
    inet 10.0.12.216/16 ...      # 自分の IP を確認

$ ping -c 4 10.0.12.216          # 上記の IP を使って確認
PING 10.0.12.216 (10.0.12.216) 56(84) bytes of data.
64 bytes from 10.0.12.216: icmp_seq=1 ttl=64 time=0.025 ms
...
4 packets transmitted, 4 received, 0% packet loss
```

> **[コンテナ制限] 外部 IP への ping は動作しない**
> `ping 8.8.8.8`（Google の公開 DNS サーバー）を実行すると 100% packet loss になります。
> Codespaces（Azure コンテナ）では外部への ICMP（Internet Control Message Protocol: ping が使う「到達確認」専用の通信規格）パケットがブロックされているためです。
>
> ```bash
> $ ping -c 4 8.8.8.8
> PING 8.8.8.8 (8.8.8.8) 56(84) bytes of data.
> (応答なし)
> 4 packets transmitted, 0 received, 100% packet loss
> ```
>
> 実際の Linux サーバーでは `ping -c 4 8.8.8.8` が成功します。

`traceroute`（パケットが外部まで届く経路を追跡するコマンド）も同じ理由でコンテナ内では全ホップが `* * *` になります。概念だけ覚えておきましょう。

```text
実機での traceroute のイメージ:
traceroute 8.8.8.8
 1  10.0.0.1     (ゲートウェイ)
 2  xxx.xxx.xxx  (プロバイダのルーター)
 3  ...
 n  8.8.8.8      (Google DNS) に到着
```

### 12-5. DNS の仕組みを理解する

`google.com` とブラウザに入力するとき、裏では「google.com」という名前を IP アドレスに変換する処理が走っています。これを**名前解決**と言い、DNS（Domain Name System）がその仕組みです。

#### 名前解決の流れ

```text
ブラウザ         /etc/hosts         DNS サーバー
   │            （連絡帳）         （電話帳サービス）
   │  ①「google.com の IP は?」
   ├─────────▶ まず確認
   │             なければ
   ├──────────────────────────▶ ②「google.com は?」
   │                                    │
   │                        ③「74.125.24.102 だよ」
   ◀──────────────────────────────────
   │
④「74.125.24.102:80 に接続！」
```

#### /etc/hosts — 自分だけの連絡帳

DNS より優先される静的な設定ファイルです。

```bash
$ cat /etc/hosts
127.0.0.1   localhost
::1         localhost ip6-localhost ip6-loopback
127.0.0.1   codespaces-bc9305
```

例えば `/etc/hosts` に `10.0.12.216 myserver` と書けば、`ping myserver` が動きます（開発環境での活用例）。

#### /etc/resolv.conf — DNS サーバーの設定

どこに問い合わせるかを設定するファイルです。

```bash
$ cat /etc/resolv.conf
nameserver 127.0.0.53
search m2fxbej512jepnsor2itp05j3d.ix.internal.cloudapp.net
nameserver 168.63.129.16
```

`127.0.0.53` は systemd-resolved（DNS キャッシュ）のアドレスです。実際には `168.63.129.16`（Azure の DNS）に問い合わせています。

#### nslookup — DNS の名前解決を確認する（シンプル）

```bash
$ nslookup google.com
Server:   127.0.0.53    ← 問い合わせた DNS サーバー
Address:  127.0.0.53#53

Non-authoritative answer:
Name:     google.com
Address:  74.125.24.102  ← google.com に対応する IP アドレスのひとつ
Name:     google.com
Address:  74.125.24.113
```

> **Non-authoritative answer（非権威応答）とは?**
> DNS キャッシュサーバーが「以前調べた結果のキャッシュ」を返していることを意味します。
> google.com を直接管理するサーバーからの回答ではありませんが、内容は同じです。

#### dig — DNS の詳細情報（上級）

```bash
$ dig google.com

; <<>> DiG 9.20.23-Debian <<>> google.com
;; ANSWER SECTION:
google.com.  59  IN  A  74.125.24.102
google.com.  59  IN  A  74.125.24.113

;; Query time: 5 msec
;; SERVER: 127.0.0.53#53(127.0.0.53)
```

| フィールド | 意味 |
|:---|:---|
| `google.com.` | 問い合わせたドメイン名（末尾の `.` はルートドメインを示す） |
| `59` | TTL（Time To Live）: あと59秒でキャッシュが切れる |
| `IN` | クラス（Internet の略、通常は IN） |
| `A` | レコードタイプ（A = IPv4 アドレス） |
| `74.125.24.102` | 結果の IP アドレス |

### 12-6. ポートとサービスを確認する

#### ポートとは何か（マンションの部屋番号）

IP アドレスは「どのマシンに届けるか」を決めます。しかし同じマシンで nginx・SSH・DNS など複数のサービスが動いています。「どのサービスに繋ぐか」を区別するのがポート番号です。

**宅配便の「住所と部屋番号」に例えると:**

```text
宅配便の伝票
┌────────────────────────────────────────┐
│ 届け先: 渋谷ビル   203 号室            │
│          ↑              ↑             │
│      IP アドレス     ポート番号        │
│   「どのマシンか」  「どのサービスか」  │
└────────────────────────────────────────┘

ブラウザが「http://10.0.12.216:80」に接続するとき:
  ① 「10.0.12.216」というマシンを見つける   ← IP アドレスの役割
  ② そのマシンの「80番」のドアをノックする  ← ポートの役割
  ③ nginx が応答して Web ページを返す
```

> IP アドレスだけでは「マンションに到着した」状態です。
> ポートを加えて初めて「どの部屋（サービス）に用があるか」が決まります。

```text
         10.0.12.216（マンション）
         ┌─────────────────────────────┐
  :80 ──▶│ 部屋  80 番 = nginx (Web)   │◀── ブラウザ
         │ 部屋  22 番 = SSH           │◀── ターミナル
  :22 ──▶│ 部屋  53 番 = DNS           │
         │ 部屋 443 番 = HTTPS (Web)   │
         └─────────────────────────────┘
```

| ポート | プロトコル | サービス |
|:---|:---|:---|
| 22 | TCP | SSH（安全なリモートログイン） |
| 53 | UDP/TCP | DNS（名前解決） |
| 80 | TCP | HTTP（Web、暗号化なし） |
| 443 | TCP | HTTPS（Web、TLS 暗号化あり） |

> **TCP と UDP の違い**
>
> - **TCP**（Transmission Control Protocol）: 届いたか確認しながら通信する信頼性重視のプロトコル。HTTP・SSH など大半の通信が使用する。
> - **UDP**（User Datagram Protocol）: 確認なしで高速に送るプロトコル。DNS の問い合わせや動画ストリーミングに適している。
>
> **TLS**（Transport Layer Security）は通信を暗号化する仕組みです。
> 以前は SSL と呼ばれており、現場では「SSL/TLS」や「SSL 証明書」と表記されることもあります（第21章で詳しく学びます）。

ポート番号は 0〜65535 まで使えます。用途によって 3 つの範囲に分かれています。

| ポート範囲 | 名称 | 特徴 |
|:---|:---|:---|
| 0〜1023 | ウェルノウンポート | **root 権限が必要**。HTTP(80)・SSH(22) など国際標準で用途が決まっている |
| 1024〜49151 | 登録済みポート | 一般ユーザーで使用可。アプリが使う（3000・8080 など） |
| 49152〜65535 | 動的ポート | OS が通信ごとに一時割り当てする（送信元ポートに使われる） |

> **自作アプリのポート番号を決めるとき**
> 0〜1023 はシステム専用なので使えません。1024〜49151 も有名サービスが使っている番号は
> 避けたほうが無難です。
> 現場でよくWebサーバーのポートとして使われるのは 8000・8080 です。
> それ以外のプログラムは  **10000 番台以降** を選ぶと衝突リスクを減らせます。

nginx が 80 番（ウェルノウンポート）を使うために root で起動する理由はここにあります（第9章・第10章の復習）。

> **ブラウザや curl ではポート番号が省略される**
> `http://localhost` と入力したとき、実際には `http://localhost:80` を意味します。
> `https://example.com` は `https://example.com:443` と同じです。
> プロトコルごとにデフォルトのポートが決まっているので、URL に書かなくても自動で補完されます。
>
> ```text
> http://localhost         → http://localhost:80    （:80 を自動補完）
> https://github.com       → https://github.com:443 （:443 を自動補完）
> http://localhost:8080    → ポートを明示（8080番で動くアプリへ直接接続）
> ```
>
> ポートを明示するのは「デフォルト以外のポートを使っているとき」だけです。

#### ss — 待ち受け中のポートを確認する

```bash
$ ss -tlnp
State  Recv-Q Send-Q Local Address:Port  Peer Address:Port  Process
LISTEN 0      511    0.0.0.0:80          0.0.0.0:*          users:(("nginx",pid=xxx))
LISTEN 0      128    0.0.0.0:22          0.0.0.0:*
LISTEN 0      4096   127.0.0.53%lo:53    0.0.0.0:*
```

オプションの意味:

| オプション | 意味 |
|:---|:---|
| `-t` | TCP のみ表示 |
| `-l` | LISTEN（待ち受け）状態のみ |
| `-n` | 数値で表示（ポート名に変換しない） |
| `-p` | どのプロセスが使っているか表示 |

`Local Address:Port` の見方:

| 表示 | 意味 |
|:---|:---|
| `0.0.0.0:80` | すべての IP アドレスの 80 番ポートで待ち受け |
| `127.0.0.1:80` | ループバック（自分自身）からのみ受け付け |
| `[::]:80` | IPv6 のすべてのアドレスで待ち受け |

nginx が起動していれば、80番ポートを確認できます。

```bash
$ service nginx status     # nginx が動いているか確認
$ ss -tlnp | grep :80
LISTEN 0   511   0.0.0.0:80   0.0.0.0:*   users:(("nginx",pid=xxx,fd=6))
```

nginx が起動していない場合は `service nginx start` で起動してから確認してください。

### 12-7. HTTP で通信してみる（curl）

`curl` はコマンドラインから HTTP 通信するツールです。ブラウザと同じようにリクエストを送り、レスポンスを受け取れます。

#### localhost の nginx に接続する

```bash
$ curl http://localhost
<!DOCTYPE html>
<html>
<head><title>Welcome to nginx!</title></head>
...
```

`-I` オプションでヘッダのみ取得できます（ページ全体を取得せず状態確認するときに便利）。

```bash
$ curl -I http://localhost
HTTP/1.1 200 OK
Server: nginx/1.24.0
Date: Sat, 30 May 2026 04:00:00 GMT
Content-Type: text/html
Content-Length: 615
```

#### 外部サイトへのアクセス

```bash
$ curl -sI https://example.com
HTTP/2 200
date: Sat, 30 May 2026 04:51:27 GMT
content-type: text/html
server: cloudflare
```

#### HTTP ステータスコードを理解する

レスポンスの1行目に含まれる3桁の数字がステータスコードです。

| コード | 意味 | よく見る場面 |
|:---|:---|:---|
| `200 OK` | 成功 | 正常にページが返ってきた |
| `301 Moved Permanently` | 恒久リダイレクト | URL が変わった |
| `403 Forbidden` | 禁止 | パーミッションで弾かれた（第11章！） |
| `404 Not Found` | 見つからない | ページが存在しない |
| `500 Internal Server Error` | サーバーエラー | サーバー側のプログラムが壊れた |
| `502 Bad Gateway` | ゲートウェイエラー | リバースプロキシの転送先が応答しない |

### コラム: リバースプロキシとファイアウォール（概念の整理）

#### リバースプロキシとは

「リバースプロキシ」は難しく聞こえますが、「受付係」のようなものだと考えれば理解しやすいです。

```text
外部からのリクエスト
       │
       ▼
┌─────────────────┐
│  nginx（受付）   │  ← 外部に公開するのは :80 だけ
│  :80 で待ち受け  │
└────────┬────────┘
         │ パスに応じて転送
         ├──▶ アプリサーバー（:3000）  例: Node.js
         └──▶ API サーバー（:8080）   例: Python Flask
```

- ブラウザからは「nginx の :80 に繋いでいる」ように見える
- 実際には nginx が後ろのアプリに「転送（proxy）」している
- 外部に公開するポートを :80 だけにできるので、内部のアプリを直接攻撃されるリスクが下がる

第20章でソースビルドした nginx を使い、リバースプロキシを自分で設定します。

#### ファイアウォールとは

ファイアウォール = 「どの部屋のドアを開けるか」を制御する仕組み。

```text
インターネット側
       │
       ▼
  ┌──────────┐
  │ ファイア   │  :80 ✓ 通す
  │ ウォール   │  :443 ✓ 通す
  │          │  :3000 ✗ ブロック（外部から直接触れない）
  └──────────┘
       │
       ▼
    nginx / アプリ
```

Debian の `ufw`（Uncomplicated Firewall）がよく使われますが、Codespaces（コンテナ）環境では期待通りに動作しないため、概念の理解にとどめます。

## よくあるミス

| ミス | エラーメッセージ例 | 正しい対処 |
|:---|:---|:---|
| `ifconfig` を使おうとする | `ifconfig: command not found` | 現代 Linux では `ip addr` が標準。`ifconfig` は古い（`net-tools` パッケージが必要） |
| `netstat` を使おうとする | `netstat: command not found` | 現代 Linux では `ss` が標準。`netstat` は `net-tools` パッケージ |
| Codespaces で `ping 8.8.8.8` | `100% packet loss` | コンテナ制限で ICMP 外向きがブロック。`ping 127.0.0.1` や自 IP で確認 |
| ポートを指定せず curl | `curl: (7) Failed to connect to localhost port 80` | nginx が起動しているか `service nginx status` で確認 |
| DNS と IP アドレスの混同 | ― | DNS は「名前→IP の変換サービス」。IP は「住所」。別の概念 |

## 類似比較

| コマンドA | コマンドB | 違い |
|:---|:---|:---|
| `ip addr` | `ifconfig` | 同じ目的だが `ip addr` が現代の標準（`ifconfig` は古い） |
| `ss -tlnp` | `netstat -tlnp` | 同じ目的だが `ss` が現代の標準（`netstat` は古い） |
| `nslookup ドメイン` | `dig ドメイン` | nslookup はシンプルな出力、dig は TTL・フラグ等の詳細出力 |
| `curl -I URL` | `curl URL` | `-I` はヘッダのみ取得、なしだとボディ（HTML 等）も取得 |
| `ping -c 4 ホスト` | `traceroute ホスト` | ping は「届くか?」の確認、traceroute は「どの経路を通るか?」の追跡 |

## 他OSとの比較

| 操作 | Linux (Debian) | Windows | macOS |
|:---|:---|:---|:---|
| IP アドレス確認 | `ip addr` | `ipconfig` | `ifconfig` または `ip addr`（Homebrew） |
| ルーティング確認 | `ip route` | `route print` | `netstat -rn` |
| 疎通確認 | `ping -c 4 ホスト` | `ping ホスト`（デフォルト4回） | `ping -c 4 ホスト` |
| ポート確認 | `ss -tlnp` | `netstat -ano` | `netstat -an` または `lsof -i` |
| DNS 確認 | `nslookup` / `dig` | `nslookup`（同名） | `nslookup` / `dig`（同名） |
| HTTP 通信 | `curl` | `curl`（PowerShell 5+で使用可） | `curl`（標準搭載） |

> **Windows との違い**
> Windows の `ping` はデフォルト4回で終了します（`-c` オプション不要）。
> Linux の `ping` は `-c` オプションを付けないと無限に送り続けます（`Ctrl+C` で停止）。

## 理解度チェック

1. IP アドレス `192.168.1.100/24` について、ネットワーク部とホスト部を答えよ。また、このネットワークには何台まで接続できるか。

<details><summary>答え</summary>

- ネットワーク部: `192.168.1`（先頭24ビット）
- ホスト部: `100`（残り8ビット）
- 接続可能台数: 2^8 - 2 = **254台**（ブロードキャスト用とネットワークアドレスを除く）

</details>

2. ブラウザで `https://github.com` を入力してからページが表示されるまでに、DNS・IP・ポートがどのように使われるか説明せよ。

<details><summary>答え</summary>

1. **DNS 解決**: ブラウザが `/etc/hosts` を確認し、なければ DNS サーバーに `github.com` の IP を問い合わせる（例: `140.82.112.4` を取得）
2. **IP で接続**: `140.82.112.4` という住所のサーバーに向けてパケットを送る
3. **ポートで識別**: `https://` なので TCP ポート **443** に接続する
4. **TLS 暗号化**: HTTPS のため TLS ハンドシェイクを行い、暗号化通信を確立
5. **HTTP リクエスト**: `GET / HTTP/1.1 Host: github.com` を送信してページを受け取る

</details>

3. `ss -tlnp | grep :80` を実行したら何も表示されなかった。考えられる原因と対処法を述べよ。

<details><summary>答え</summary>

原因: **nginx が起動していない**ため、80番ポートを誰も待ち受けていない。

対処:

```bash
$ service nginx status   # 状態を確認
$ service nginx start    # 起動する
$ ss -tlnp | grep :80    # 再確認
```

</details>

4. `/etc/hosts` に `10.0.12.216 myapp` と追記した場合、`curl http://myapp` は何をするか説明せよ。

<details><summary>答え</summary>

1. `curl` が `myapp` という名前を解決しようとする
2. `/etc/hosts` を確認し、`myapp` → `10.0.12.216` というマッピングを発見
3. DNS サーバーには問い合わせず、`10.0.12.216:80` に HTTP リクエストを送信
4. そのマシンの 80 番ポートで nginx が動いていれば応答が返ってくる

`/etc/hosts` は DNS より優先されるため、DNS サーバーを経由せずに解決できる。
開発環境でのテストや、本番のドメインが取得前の動作確認に使われる。

</details>

5. nginx をリバースプロキシとして使う場合、ポート 3000 で動くアプリサーバーはなぜ外部から直接アクセスできないようにすべきか？

<details><summary>答え</summary>

セキュリティ上の理由から:

- ポート 3000 を直接公開すると、ファイアウォールの設定次第で外部から直接攻撃される
- nginx を経由させることで「認証チェック」「レート制限」「ログ記録」を一元管理できる
- 外部に公開するポートを 80 と 443 だけに絞ることで、攻撃対象を減らせる（最小公開の原則）
- アプリサーバーを `127.0.0.1:3000`（ループバックのみ）で起動すれば、同じマシンの nginx からしかアクセスできなくなる

</details>
