# 第0章: Linux の歴史・ディストリビューションとは

## 前提知識

なし（この章が出発点です）

---

## 概要

Linux はサーバー・クラウド・コンテナの世界で圧倒的なシェアを誇る OS です。
しかし「Linux」という言葉は「カーネル」「ディストリビューション」「OS」など複数の意味で使われるため、最初に整理しておく必要があります。
この章では Linux の誕生の歴史を辿りながら、カーネル・OS・ディストリビューション・Webブラウザの違いという基本概念を押さえ、この講座を通じて使う Debian Linux がなぜ選ばれているかを理解します。

---

## 手順

### 0-1. OS・カーネル・ディストリビューション・Webブラウザの違い

新人エンジニアが最初に混乱しがちな4つの概念を「ソフトウェアの層（レイヤー）」として整理します。

```text
┌──────────────────────────────────────────────────┐
│  アプリケーション層                                 │
│  例: Webブラウザ (Firefox / Chrome)                │
│      テキストエディタ、nginx、Python プログラム       │
├──────────────────────────────────────────────────┤
│  OS（オペレーティングシステム）                       │
│  = カーネル + ユーザーランド                         │
│  ┌─────────────────────────────────────────┐      │
│  │ カーネル（Linux kernel）                  │     │
│  │ ・プロセス管理（どのプログラムをいつ動かすか）  │     │
│  │ ・メモリ管理（誰がどのメモリを使うか）         │     │
│  │ ・ファイルシステム（ファイルの読み書き）       │     │
│  │ ・デバイスドライバ（ハードを制御）            │     │
│  └─────────────────────────────────────────┘     │
│  ユーザーランド: bash, ls, apt, systemd 等          │
├──────────────────────────────────────────────────┤
│  ハードウェア（CPU・メモリ・ストレージ・NIC）          │
└──────────────────────────────────────────────────┘
```

> **「ユーザーランド」とは?** カーネルより上の層で動くプログラム群の総称です。
> `bash`（コマンドを受け付けるシェル）、`ls`（ファイル一覧を表示するコマンド）、
> `apt`（パッケージ管理ツール）などがユーザーランドに属します。
> カーネルが「OS の核心」なら、ユーザーランドは「ユーザーが実際に触れる部分」です。

| 用語 | 正確な定義 | 例 |
|:---|:---|:---|
| **カーネル** | ハードウェアを直接制御する OS の核心部分。単体では使えない | Linux カーネル 6.x |
| **OS** | カーネル + 基本ユーザーランド（シェル等）の総称。使える状態の基盤 | GNU/Linux |
| **ディストリビューション** | OS（カーネル込み）+ パッケージ管理 + 設定ツールをまとめた「完成品」。**ディストリビューション = 完全な OS** | Ubuntu / Debian / RHEL |
| **Webブラウザ** | OS の上で動くアプリケーションの1つ。OS ではない | Firefox / Chrome / Safari |

**「ディストリビューション」と「OS」は同じ？**

→ **ほぼ同じ意味**で使われます。「Ubuntu を使っています」は「Ubuntu という Linux OS を使っています」と同じです。
ただし「Linux カーネル」単体は使えないため、日常で「Linux OS」と言えば必ずいずれかのディストリビューションを指します。

```text
Linux カーネル単体  ≠  完全な OS（使えない）
ディストリビューション  =  完全な OS（使える）
```

> **ポイント:** Webブラウザは OS ではなくアプリケーションです。Firefox は Linux の上でも Windows の上でも動きます。「OS を変えたらブラウザも変わった」ということはありません。

### 0-2. Unix の誕生（1969年）

Linux を理解するには、その親である Unix を知る必要があります。

- **1969年**、AT&T ベル研究所の **Dennis Ritchie** と **Ken Thompson** が Unix を開発
- もともとは研究者が快適に使えるシステムを作る目的で始まった
- プログラミング言語 **C言語** で書かれており、異なるハードウェアへの移植が容易（**ポータビリティ**: 同じコードを別のCPU・マシンでも動かせる性質）
- 「1つのプログラムは1つのことをうまくやる」という Unix 哲学が、現在の Linux コマンド設計にも受け継がれている

---

### 0-3. Linux カーネルの誕生（1991年）

- **1991年**、フィンランドのヘルシンキ大学の学生だった **Linus Torvalds** が趣味で開発を開始
- 「MINIX（教育用 Unix）より良い OS が欲しかった」という動機
- **GPL（GNU General Public License）** ライセンスで公開 → 世界中の開発者が参加できる仕組みに
  > **GNU とは?** "GNU's Not Unix" の頭字語。Richard Stallman が 1983年に立ち上げた、誰でも自由に使える OS とツール群を作るプロジェクト名です。bash・ls・grep など普段使うコマンドの多くは GNU が作っています。
- 現在も毎週新しいバージョンがリリースされ、世界規模のコントリビュータ（開発に貢献する参加者）が開発を継続している

実際に、今この Codespaces で動いている Linux カーネルのバージョンを確認してみましょう:

```bash
# 現在動いている Linux カーネルのバージョンを確認する
$ uname -r
6.8.0-1052-azure
```

> **「6.8.0-1052-azure」の読み方:**
> `6.8.0` がカーネルのバージョン番号。`-1052-azure` は GitHub Codespaces（Azure）向けのビルド番号です。
> この数字は Linus Torvalds のチームが毎週改良・公開し続けているものと同じ系譜です。
> あなたの環境では異なる番号が表示されることがありますが、`6.x` 系であれば問題ありません。

### コラム: Linux という名前の由来

「Linux」という名前は、実は **Linus 本人が付けたものではありません**。

- Linus Torvalds はもともと「**Freax**」と命名するつもりだった
  （free + freak + Unix の x を組み合わせた造語）
- 1991年、ヘルシンキ大学の FTP サーバー（File Transfer Protocol: ファイル転送の通信規格を使ったサーバー）に公開したとき、サーバー管理者だった **Ari Lemmke** が勝手にディレクトリ名を `linux` に変更してしまった
- Linus 本人は当初気に入っていなかったが、そのまま世界中に広まった
- 結果として、Linus の名前 + Unix を組み合わせた「Linux」が定着した

### 0-4. ディストリビューションとは（= 完成した Linux OS）

Linux カーネルだけではシステムとして使えません。そこで登場するのがディストリビューションです。

```text
Linux カーネル
  +
GNU ツール群（bash, ls, grep 等）
  +
パッケージ管理システム（apt, dnf, pacman 等）
  +
設定ツール・ドキュメント
  =
ディストリビューション（= 使えるOS）
```

**なぜこんなに多くのディストリビューションが存在するのか？**

| 理由 | 例 |
|:---|:---|
| 用途の違い | デスクトップ用・サーバー用・コンテナ用 |
| サポートモデルの違い | 商用サポートあり（RHEL）vs コミュニティ主導（Debian） |
| 哲学・思想の違い | 安定性重視（Debian）vs 最新技術重視（Arch Linux） |
| 軽量化・特化 | コンテナ特化（Alpine Linux） |

> **比喩:** カーネルは「エンジン」、ディストリビューションは「完成した車（= OS）」です。エンジン単体では走れませんが、車体・タイヤ・ハンドルと組み合わせると走れるようになります。

---

### 0-5. 代表的なディストリビューション比較

| ディストリビューション | 系統 | パッケージ管理 | 主な用途 |
|:---|:---|:---|:---|
| **Debian** | Debian 系 | apt / dpkg | サーバー・開発基盤（**本講座の環境**） |
| Ubuntu | Debian 系（派生） | apt / dpkg | デスクトップ・クラウド |
| RHEL | Red Hat 系 | dnf / rpm | 商用サーバー（有料サポート） |
| CentOS / AlmaLinux / Rocky | Red Hat 系（互換） | dnf / rpm | 商用互換サーバー |
| Fedora | Red Hat 系（上流） | dnf / rpm | 最新技術の検証・開発 |
| Arch Linux | 独立系 | pacman | 上級者・カスタマイズ重視 |
| Alpine Linux | 独立系 | apk | コンテナ特化（軽量・セキュリティ重視） |

> **本講座について:** GitHub Codespaces 上の **Debian** を使います。Debian は Ubuntu の親にあたるディストリビューションであり、サーバー用途で広く使われています。

### 0-6. オープンソースと GPL ライセンス

Linux カーネルを世界規模の協力開発が可能にした鍵は、ライセンスにあります。

**オープンソース ≠ 無料**

「オープンソース」とは価格（無料・有料）の話ではなく、**自由（Freedom）** の話です：

- ソースコードを読む自由
- 改変する自由
- 再配布する自由

> RHEL（Red Hat Enterprise Linux）はオープンソースですが、商用サポートは**有料**です。

**GPL（GNU General Public License）の核心**

Linux カーネルは **GPLv2** というライセンスで公開されています。

| 条件 | 内容 |
|:---|:---|
| 使用 | 自由に使える |
| 改変 | 自由に改変できる |
| 配布 | GPL のままで配布しなければならない |
| **ソースコード公開** | **GPL ソフトウェアを改変して配布する場合、改変後のソースコードも公開しなければならない** |

この「ソースコード公開義務」は**コピーレフト**と呼ばれます。
著作権（コピーライト）をもじった造語で、「もらった自由を次の人にも渡す義務」を意味します。
世界中のエンジニアが Linux の改善に貢献する仕組みを生み出した鍵です。

> **「コピーレフト」とは?**
> 「改変して配布する場合は、改変後のソースコードも同じライセンス（GPL）で公開しなければならない」という性質を指す概念です。
> 自分が受け取った自由（使用・改変・配布）を、次に受け取る人にも保障しなければなりません。

### コラム: なぜ今 Linux を学ぶのか

現代のバックエンドエンジニアがサーバーを扱う場面では、ほぼ必ず Linux が登場します。

| 分野 | Linux との関係 |
|:---|:---|
| クラウド（AWS / GCP / Azure） | サーバーインスタンスの大部分が Linux |
| Docker / Kubernetes | Linux カーネルの namespace（プロセス隔離の仕組み）・cgroups（CPU/メモリ使用量の制限機能）の上に成り立つ |
| GitHub Actions / CI・CD | 実行環境のほとんどが Ubuntu |
| 組み込み・IoT | 家電・ルーター・自動車にも Linux が搭載 |

**この講座のゴール:** 第19章で Nginx を手動ビルドし、第21章で Docker が同じことを自動でやってのける様子を見ることで、「Linux を理解したうえでコンテナを使いこなすエンジニア」になることを目指します。

## よくあるミス

| よくある誤解 | 正しい理解 |
|:---|:---|
| 「Linux = Ubuntu のこと」 | Linux はカーネル名。Ubuntu はそのカーネルを使ったディストリビューション（OS）の1つ |
| 「オープンソース = 無料」 | 自由（Freedom）と無料（Free of charge）は別概念。RHEL はオープンソースだが有料 |
| 「CentOS は今でも RHEL の無償版」 | CentOS 8 は2021年末サポート終了。後継は AlmaLinux / Rocky Linux |
| 「Alpine Linux は軽いから性能が低い」 | Alpine はセキュリティと軽量性を重視した設計。コンテナ用途では標準的な選択肢 |
| 「ディストリビューションと OS は別物」 | ディストリビューション = 完全な OS。この2つはほぼ同義 |

## 類似比較

各ディストリビューションのパッケージ管理コマンドは、第4章で詳しく学びます。ここでは対応関係だけ把握しておきましょう。

| 操作 | Debian / Ubuntu | RHEL / CentOS | Arch Linux | Alpine Linux |
|:---|:---|:---|:---|:---|
| パッケージインストール | `apt install` | `dnf install` | `pacman -S` | `apk add` |
| リポジトリ情報更新 | `apt update` | `dnf check-update` | `pacman -Sy` | `apk update` |
| パッケージ削除 | `apt remove` | `dnf remove` | `pacman -R` | `apk del` |
| パッケージ形式 | `.deb` | `.rpm` | `.pkg.tar.zst` | `.apk` |
| 低レベルツール | `dpkg` | `rpm` | — | — |

## 他OSとの比較

| 観点 | Linux（Debian） | Windows | macOS |
|:---|:---|:---|:---|
| カーネル | Linux カーネル（オープンソース・GPLv2） | Windows NT（クローズドソース） | Darwin / XNU（BSD 派生、一部公開） |
| ソースコード | GPL で全公開 | 非公開 | 一部公開（Darwin は OSS） |
| 主な用途 | サーバー・コンテナ・組み込み | デスクトップ・ゲーム・業務 | デスクトップ・開発 |
| パッケージ管理 | apt / dpkg | winget / Chocolatey / MSI インストーラ | Homebrew / App Store |
| 価格 | 基本無料（商用サポートは有料） | 有料（OEM は端末価格に含む） | macOS 本体は無料（Apple ハードウェアが有料） |
| シェア（サーバー） | 約 80〜90% | 少数 | ほぼなし |

## 理解度チェック

1. カーネル・OS・ディストリビューション・Webブラウザを「ソフトウェアの層」として説明してください。それぞれどの層に位置しますか？

<details><summary>答え</summary>

```text
アプリケーション層:  Webブラウザ（Firefox / Chrome など）
OS 層:              ディストリビューション = カーネル + ユーザーランド
  └ カーネル層:      Linux カーネル（ハードウェアを直接制御）
ハードウェア層:      CPU・メモリ・ストレージ
```

- **カーネル**: OS の核心。ハードウェアを直接制御するが、単体では使えない
- **OS**: カーネル + bash などの基本ツールを含む、使える状態の基盤
- **ディストリビューション**: OS（カーネル込み）+ パッケージ管理 + 設定ツール = 完全な OS
- **Webブラウザ**: OS の上で動くアプリケーション。OS ではない

</details>

2. 「Linux」と「Linux ディストリビューション」の違いを説明してください。

<details><summary>答え</summary>

- **Linux** = Linux カーネルそのもの。カーネル単体では OS として使えない
- **Linux ディストリビューション** = Linux カーネル + GNU ツール群 + パッケージ管理システムをまとめた「完成した OS」

日常的に「Linux を使っている」と言う場合、実際には Ubuntu や Debian などのディストリビューションを指している。

</details>

3. Debian 系と Red Hat 系では、パッケージのインストールコマンドが異なります。それぞれ答えてください。

<details><summary>答え</summary>

| 系統 | パッケージインストールコマンド |
|:---|:---|
| Debian 系（Debian / Ubuntu） | `apt install パッケージ名` |
| Red Hat 系（RHEL / CentOS / Fedora） | `dnf install パッケージ名` |

パッケージ形式も異なる: Debian 系は `.deb`、Red Hat 系は `.rpm`

</details>

4. GPL ライセンスでは、ソースコードを改変して配布する場合にどのような義務が生じますか？

<details><summary>答え</summary>

改変後のソースコードも **GPL ライセンスのまま公開しなければならない**。

これを「コピーレフト」と呼ぶ。自分が改変して配布した場合、受け取った人も同じ自由を持たなければならない、という考え方。Linux カーネルはこの仕組みにより、世界中のエンジニアが改善を積み重ねる協力開発を実現している。

</details>

5. CentOS 8 がサポート終了した後、RHEL 互換として登場したディストリビューションを2つ答えてください。

<details><summary>答え</summary>

- **AlmaLinux** — CloudLinux 社が主導するコミュニティプロジェクト
- **Rocky Linux** — CentOS の創設者 Gregory Kurtzer が立ち上げたプロジェクト

どちらも RHEL とバイナリ互換（コンパイル済みのプログラムをそのまま動かせるという互換性）を目指しており、CentOS 8 の後継として広く採用されている。

</details>

6. なぜクラウド（AWS・GCP・Azure）のサーバーに Linux が多く採用されているか、理由を2つ以上挙げてください。

<details><summary>答え</summary>

主な理由（例）:

1. **ライセンスコストがかからない** — Windows Server ライセンスと異なり、Linux は無料で利用できる（商用サポートを除く）
2. **安定性・信頼性が高い** — 長期間無停止で動作する実績があり、サーバー用途に適している
3. **軽量で資源効率が良い** — GUI なしのサーバー向けインストールが可能で、メモリ・CPU を節約できる
4. **コンテナ技術との親和性** — Docker / Kubernetes は Linux カーネルの namespace・cgroups 機能を直接利用する
5. **豊富なエコシステム** — サーバー向けソフトウェアのほとんどが Linux を第一サポートとして開発されている

</details>
