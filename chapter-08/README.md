# 第8章: Locale・Timezone を設定する

## 前提知識

この章を始める前に、以下の章を完了していること:

- 第6章: シェル環境をカスタマイズする
- 第7章: 環境変数・入力補完・カラー表示

---

## 概要

OS の**ロケール**（言語・文字コード・日付書式の設定）と**タイムゾーン**（時刻の基準となる地域設定）を確認・変更します。
ロケールの設定は文字化け防止に、タイムゾーンの設定は nginx アクセスログのタイムスタンプに直結します。
また、コラムとして Linux の核心部分である**カーネル**とは何かを `uname -r` で確認しながら学びます。

---

## 手順

### 8-1. ロケールとは何か

**ロケール**とは、「どの言語・地域・文字コードに従って動作するか」を OS に伝える設定です。
`LANG=ja_JP.UTF-8` のように `言語_地域.文字コード` の形式で表します。

#### 現在のロケール設定を確認する

```bash
$ locale
LANG=
LANGUAGE=
LC_CTYPE="POSIX"
LC_NUMERIC="POSIX"
LC_TIME="POSIX"
LC_COLLATE="POSIX"
LC_MONETARY="POSIX"
LC_MESSAGES="POSIX"
LC_PAPER="POSIX"
LC_NAME="POSIX"
LC_ADDRESS="POSIX"
LC_TELEPHONE="POSIX"
LC_MEASUREMENT="POSIX"
LC_IDENTIFICATION="POSIX"
LC_ALL=
```

`LANG=` が空の状態は「POSIX ロケール」が使われていることを意味します。
第7章で `echo $LANG` が空だった理由がここで明確になります。

#### ロケール変数の役割

| 変数 | 役割 | 設定例 |
|:---|:---|:---|
| `LANG` | デフォルトのロケール。他の `LC_*` が未設定のときのフォールバック | `ja_JP.UTF-8` |
| `LC_TIME` | 日付・時刻の表示形式 | `ja_JP.UTF-8` |
| `LC_MESSAGES` | システムメッセージ（エラー文など）の言語 | `ja_JP.UTF-8` |
| `LC_ALL` | 全 `LC_*` を強制的に一括上書き（最優先） | `C` |
| `LANGUAGE` | 翻訳メッセージの優先言語リスト | `ja:en` |

> **`LANG` と `LC_ALL` の違い**
> `LANG` は「他の LC_* が設定されていないときのデフォルト」です。
> `LC_ALL` は `LANG` を含む全変数を強制上書きします。スクリプト内で `LC_ALL=C` と書くと、どの環境でも英語・ASCII 出力に固定できます。

#### POSIX ロケールとは

`C` または `POSIX` は、英語・ASCII のみを扱う最小限のロケールです。
日本語を含むマルチバイト文字（**マルチバイト文字**: 1文字を表すのに複数バイトを使う文字。日本語・中国語・韓国語など）は正しく扱えませんが、スクリプトや自動化処理で「環境に依存しない安定した出力」が必要なときに使います。

### 8-2. ロケールを確認・設定する

#### 利用可能なロケール一覧を確認する

```bash
$ locale -a
C
C.utf8
POSIX
en_US.utf8
```

初期状態では `ja_JP.UTF-8` はまだ生成されていません。

> **補足:** 環境によっては、初期状態でも `ja_JP.utf8` が表示される場合があります。
> これは `locale-archive` にあらかじめロケールデータが収録されているためです。
> その場合でも、`/etc/locale.gen` を管理することで生成するロケールを明示的に制御できるため、
> 以降の手順は同様に実施してください。

#### `/etc/locale.gen` を編集して日本語ロケールを有効化する

ロケールを追加するには、まず `/etc/locale.gen` の**コメントアウト**（`#` で始まる行を「無効化」する記法。`#` を取り除くと有効になる）を解除し、その後 `locale-gen` で生成します。

> **なぜ vim で編集しないのか?**
> 1行だけ `#` を取り除くだけなので、`sed`（ストリームエディタ）コマンドで自動化する方が確実です。
> `sed` は第16章で詳しく学びますが、ここでは「指定パターンを検索して置換するコマンド」として使います。
> vim で手動編集しても結果は同じです。

```bash
# /etc/locale.gen の ja_JP.UTF-8 行のコメントアウトを解除する
$ sudo sed -i 's/# ja_JP.UTF-8 UTF-8/ja_JP.UTF-8 UTF-8/' /etc/locale.gen

# 有効化されたことを確認する
$ grep "ja_JP.UTF-8" /etc/locale.gen
ja_JP.UTF-8 UTF-8
```

> **`locale-gen` に引数を渡してはいけない**
> `sudo locale-gen ja_JP.UTF-8` のように引数付きで実行しても動作しません。
> 必ず先に `/etc/locale.gen` を編集してから、引数なしで `sudo locale-gen` を実行してください。

#### ロケールを生成する

```bash
$ sudo locale-gen
Generating locales (this might take a while)...
  en_US.UTF-8... done
  ja_JP.UTF-8... done
Generation complete.
```

```bash
# 生成されたか確認する
$ locale -a
C
C.utf8
POSIX
en_US.utf8
ja_JP.utf8
```

`ja_JP.utf8` が追加されました。

> **表示名の揺れについて**
> `locale -a` では `ja_JP.utf8`（小文字）と表示されますが、実体は `ja_JP.UTF-8` と同じです。
> `update-locale` に渡す値は `LANG=ja_JP.UTF-8`（大文字）を使います。

#### デフォルトロケールを設定する

```bash
$ sudo update-locale LANG=ja_JP.UTF-8

# 設定ファイルを確認する
$ cat /etc/default/locale
#  File generated by update-locale
LANG=ja_JP.UTF-8
```

#### 新しいターミナルで反映を確認する

`update-locale` は `/etc/default/locale` を書き換えるだけで、起動済みのシェルには反映されません。
新しいターミナルを開いてから確認してください。

```bash
# 新しいターミナルを開いてから実行する
$ locale
LANG=ja_JP.UTF-8
LC_CTYPE="ja_JP.UTF-8"
LC_NUMERIC="ja_JP.UTF-8"
LC_TIME="ja_JP.UTF-8"
LC_COLLATE="ja_JP.UTF-8"
LC_MONETARY="ja_JP.UTF-8"
LC_MESSAGES="ja_JP.UTF-8"
LC_PAPER="ja_JP.UTF-8"
LC_NAME="ja_JP.UTF-8"
LC_ADDRESS="ja_JP.UTF-8"
LC_TELEPHONE="ja_JP.UTF-8"
LC_MEASUREMENT="ja_JP.UTF-8"
LC_IDENTIFICATION="ja_JP.UTF-8"
LC_ALL=
```

`LANG=ja_JP.UTF-8` に変わりました。

### 8-3. タイムゾーンを確認・変更する

**タイムゾーン**とは、世界標準時（UTC: Coordinated Universal Time、協定世界時）との時差を定めた地域設定です。
日本は UTC+9 で、`Asia/Tokyo` タイムゾーンを使います。

#### 現在のタイムゾーンを確認する

Linux はタイムゾーン情報を `/etc/localtime` というシンボリックリンクで管理します。
リンク先を見ることで現在の設定がわかります。

```bash
$ readlink /etc/localtime
/usr/share/zoneinfo/Etc/UTC

$ date
Wed May 27 13:00:59 UTC 2026
# ※ 表示される時刻は実行時刻により異なります
```

#### 利用可能なタイムゾーンを確認する

```bash
$ ls /usr/share/zoneinfo/Asia/ | grep -E '^(Tokyo|Seoul|Singapore|Shanghai)'
Seoul
Shanghai
Singapore
Tokyo
```

#### タイムゾーンを Asia/Tokyo に変更する

**シンボリックリンク**とは、Windows のショートカット（.lnk ファイル）に相当する仕組みで、あるファイルパスから別のファイルへの参照を張る機能です。`/etc/localtime` はタイムゾーンデータへのシンボリックリンクになっています。

シンボリックリンクを張り替えることでタイムゾーンを変更します。

```bash
$ sudo ln -sf /usr/share/zoneinfo/Asia/Tokyo /etc/localtime

# 変更後のリンク先を確認する
$ readlink /etc/localtime
/usr/share/zoneinfo/Asia/Tokyo

# 日時表示が JST（UTC+9）に変わったことを確認する
$ date
Wed May 27 22:00:59 JST 2026
```

`JST` と `+0900` が表示されれば設定完了です。この設定は章の終わりまで保持します。

#### Nginx との接点: タイムゾーンがアクセスログに影響する

> 第20章でソースビルドした nginx のアクセスログ（`/var/log/nginx/access.log`）には、OS のタイムゾーンに基づいたタイムスタンプが記録されます。
> `Asia/Tokyo` を設定しておくと、ログが JST（日本標準時）で記録されるため、アクセス時刻の読み取りが容易になります。
>
> ```text
> 192.168.1.1 - - [27/May/2026:22:00:59 +0900] "GET / HTTP/1.1" 200 612 ...
>                                        ↑ +0900 は JST（UTC+9）を意味する
> ```

#### 一時的にタイムゾーンを変更する

第7章で学んだ `VAR=value command` 形式を使うと、設定ファイルを変えずに1コマンドだけ別のタイムゾーンで実行できます。

```bash
# TZ 変数を一時的に設定して date を実行する
$ TZ=Asia/Tokyo date
Wed May 27 22:00:59 JST 2026

# 現在のシェルのタイムゾーンは変わっていない
$ date
Wed May 27 22:00:59 JST 2026
```

> ここでは `sudo ln -sf` で既に JST に変更済みのため、両方とも JST になります。
> UTC 環境で試す場合は `TZ=America/New_York date` などで効果を確認できます。

#### 実サーバーでの標準コマンド: timedatectl

> **[コンテナ制限] Codespaces 環境での `timedatectl`**
> Codespaces のコンテナ環境では PID 1 が systemd（Linux の起動・サービス管理を担うプログラム。Windows のサービスコントロールマネージャーに相当）ではないため、`timedatectl` を実行すると以下のエラーになります。
>
> ```text
> System has not been booted with systemd as init system (PID 1). Can't operate.
> Failed to connect to system scope bus via local transport: Host is down
> ```
>
> 実際の Linux サーバーや VM では以下のように使えます。Codespaces ではシンボリックリンク方式（`sudo ln -sf`）を使ってください。

実サーバー（systemd が動作している環境）での操作:

```bash
# タイムゾーンと NTP の状態を確認する
$ timedatectl
               Local time: Wed 2026-05-27 22:00:59 JST
           Universal time: Wed 2026-05-27 13:00:59 UTC
                 RTC time: n/a
                Time zone: Asia/Tokyo (JST, +0900)
System clock synchronized: yes
              NTP service: active
          RTC in local TZ: no

# 利用可能なタイムゾーンを絞り込む
$ timedatectl list-timezones | grep Asia

# タイムゾーンを変更する
$ sudo timedatectl set-timezone Asia/Tokyo
```

### 8-4. コラム: カーネルとは

**カーネル**とは、Linux OS の核心部分です。アプリケーションとハードウェアの橋渡し役として、CPU・メモリ・ディスクなどのリソースを管理します。
Java で言えば JVM に近い存在で、アプリケーションが直接ハードウェアを操作する代わりにカーネルを経由することで、安全で移植性のある動作が保証されます。

#### カーネルバージョンを確認する

```bash
$ uname -r
6.8.0-1052-azure

$ uname -a
Linux codespaces-xxxxxx 6.8.0-1052-azure #58~22.04.1-Ubuntu SMP Thu Mar 26 05:02:21 UTC 2026 x86_64 GNU/Linux
```

#### カーネルの主な役割

| 役割 | 内容 |
|:---|:---|
| プロセス管理 | CPU 時間を各プロセスに配分する（マルチタスク） |
| メモリ管理 | プロセスごとのメモリ領域を管理・保護する |
| デバイスドライバ | HDD・NIC などのハードウェアを抽象化してアプリから使えるようにする |
| システムコール | アプリ（ユーザー空間）とカーネル（カーネル空間）の橋渡し |

> **Codespaces のカーネル: `6.8.0-1052-azure`**
> `-azure` サフィックスは Microsoft Azure のハイパーバイザー向けに最適化された Ubuntu カーネルを意味します。
> Codespaces はコンテナ上で動作しており、コンテナ内から見えるカーネルはホスト OS（Azure VM）のカーネルを共有しています。
> そのため `uname -r` の値はコンテナを再作成しても変わりません。

## よくあるミス

| ミス | エラーメッセージ例 | 正しい対処 |
|:---|:---|:---|
| `locale-gen` を実行せずに `update-locale` を実行した | `*** update-locale: Error: invalid locale settings: LANG=ja_JP.UTF-8`（必ずエラーになり設定が保存されない） | `/etc/locale.gen` を編集してから `sudo locale-gen` を先に実行する |
| `update-locale` 後に同じターミナルで `locale` を確認した | `LANG=` が空のまま（変更前の状態が表示される） | 新しいターミナルを開いてから `locale` を実行する |
| タイムゾーン変更後に `date` を確認しなかった | （気づかずに誤ったタイムゾーンのまま運用する） | `date` で `JST` 表示を確認する |
| Codespaces で `timedatectl` を実行した | `System has not been booted with systemd as init system (PID 1). Can't operate.` | `sudo ln -sf /usr/share/zoneinfo/Asia/Tokyo /etc/localtime` を使う |

## 類似比較

タイムゾーン変更方法の比較:

| 方法 | コマンド | 有効範囲 | Codespaces |
|:---|:---|:---|:---|
| タイムゾーン変更（永続） | `sudo ln -sf /usr/share/zoneinfo/Asia/Tokyo /etc/localtime` | システム全体 | 動作する |
| タイムゾーン変更（永続・systemd） | `sudo timedatectl set-timezone Asia/Tokyo` | システム全体 | **動作しない** |
| タイムゾーン変更（一時・1コマンドのみ） | `TZ=Asia/Tokyo date` | そのコマンドのみ | 動作する |
| ロケール変更（永続） | `sudo update-locale LANG=ja_JP.UTF-8` | システム全体（次ログイン以降） | 動作する |

| 操作 | Linux (Debian) | Windows | macOS |
|:---|:---|:---|:---|

## 理解度チェック

1. `LANG` と `LC_ALL` の違いは何か？

<details><summary>答え</summary>

`LANG` は他の `LC_*` 変数が未設定のときのデフォルト値として機能します。
`LC_ALL` は `LANG` を含む全 `LC_*` 変数を強制的に上書きします。

スクリプト内で環境に依存しない英語出力が必要な場合、`LC_ALL=C command` のように使います。

</details>

2. `sudo locale-gen ja_JP.UTF-8`（引数付き）ではなく、引数なしの `sudo locale-gen` を使う理由は何か？

<details><summary>答え</summary>

`locale-gen` は `/etc/locale.gen` の内容に従ってロケールを生成します。
引数付きで実行しても動作しないため、必ず先に `/etc/locale.gen` を編集してコメントアウトを解除し、引数なしで `sudo locale-gen` を実行します。

正しい手順:

```bash
sudo sed -i 's/# ja_JP.UTF-8 UTF-8/ja_JP.UTF-8 UTF-8/' /etc/locale.gen
sudo locale-gen
```

</details>

3. `sudo update-locale LANG=ja_JP.UTF-8` を実行した後、同じターミナルで `locale` を確認しても `LANG=` が空のままなのはなぜか？

<details><summary>答え</summary>

`update-locale` は `/etc/default/locale` を書き換えるだけで、起動済みのシェルには反映されません。
新しいシェルはログイン時に `/etc/default/locale` を読み込むため、新しいターミナルを開いてから `locale` を実行すると `LANG=ja_JP.UTF-8` が表示されます。

</details>

4. Codespaces で `timedatectl` が動作しない理由は何か？代替コマンドは何か？

<details><summary>答え</summary>

`timedatectl` は systemd（Linux の init システム）と連携するコマンドです。
Codespaces のコンテナ環境では PID 1 が systemd ではないため、以下のエラーになります。

```text
System has not been booted with systemd as init system (PID 1). Can't operate.
Failed to connect to system scope bus via local transport: Host is down
```

Codespaces での代替コマンド:

```bash
sudo ln -sf /usr/share/zoneinfo/Asia/Tokyo /etc/localtime
date    # JST 表示を確認
```

</details>

5. `uname -r` で表示される `6.8.0-1052-azure` の `-azure` サフィックスは何を意味するか？

<details><summary>答え</summary>

`-azure` は Microsoft Azure のハイパーバイザー向けに最適化された Ubuntu カーネルであることを示します。
Codespaces は Azure VM 上で動作するコンテナであり、コンテナはホスト OS のカーネルを共有します。
そのため `uname -r` の出力はホスト Azure VM のカーネルバージョンが表示されます。

</details>
